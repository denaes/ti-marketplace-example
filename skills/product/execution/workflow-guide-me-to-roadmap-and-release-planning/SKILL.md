# Roadmap and release planning

Produce an outcome-oriented roadmap, prioritize initiatives or features, and draft release notes. Use when planning a quarter, a release, or communicating what is shipping.

## When to Use

- You need a **roadmap** (outcome-based or feature-based) for a product area or quarter
- You want to **prioritize** features or initiatives with a clear framework
- You are drafting **release notes** for an upcoming or past release

## Prerequisites

- Input: list of initiatives/features, target timeframe, and (if release notes) list of changes or epic summaries
- Skills: `skills/product/strategy/roadmap-planning/SKILL.md`, `skills/product/execution/outcome-roadmap/SKILL.md`, `skills/product/execution/prioritization-frameworks/SKILL.md`, `skills/product/discovery/prioritize-features/SKILL.md`, `skills/product/execution/release-notes/SKILL.md`, `skills/product/execution/deliver-release-notes/SKILL.md`

## Step 1: Clarify scope and output

1. **Confirm with the user:** Roadmap only, prioritization only, release notes only, or combination?
2. **Inputs:** Do they have a backlog list, epic list, or release scope? Time horizon (quarter, half, release)?
3. If unclear, ask per `standards/conventions.md`.

## Step 2: Prioritization (if applicable)

1. Read **prioritization-frameworks** and **prioritize-features**.
2. Apply a chosen framework (e.g. value vs effort, RICE, MoSCoW) to the list they provided. Produce a prioritized list with short rationale.
3. Share the ranked list with the user; adjust if they want different criteria.

## Step 3: Roadmap (if applicable)

1. Read **outcome-roadmap** and **roadmap-planning**.
2. Produce a roadmap view: outcomes or themes on a timeline (now / next / later), with the prioritized items placed. Keep it to one page or a short table unless they ask for more.
3. Optionally tie to phases (Alpha/Beta/GA) if the user uses them.

## Step 4: Release notes (if applicable)

1. Read **release-notes** and **deliver-release-notes**.
2. Draft release notes from the list of changes (or epic summaries). Use the structure from deliver-release-notes (sections, audience, tone).
3. Output as markdown; user can paste into Confluence, email, or product.

## Step 5: Output and next steps

1. Write each output to a file (e.g. `workspace/_notes/roadmaps/YYYY-MM-DD_roadmap.md`, `YYYY-MM-DD_release-notes.md`).
2. Tell the user how to revise or reuse for the next cycle.

## References

- Conventions: `standards/conventions.md`
