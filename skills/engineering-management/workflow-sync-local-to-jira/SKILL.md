# Push Stories to Jira

This workflow takes generated story files from an **epic folder** and creates (or updates) Jira issues using the Atlassian MCP server. The epic folder is either a **phase folder** (e.g. `workspace/projects/<slug>/eng/alpha/`, `eng/beta/`, `eng/ga/`, `eng/fast-follow/`) or a single epic folder (`eng/epic/`, `epics/`). It supports pushing one phase epic, a single story, or a subset of stories. **When the epic lives in `workspace/projects/`**, after push the folder and story files are renamed with the Jira ticket ID prefix (see Step 8).

## Prerequisites

- Story files generated by `/generate-epics-and-stories` in a PRD output folder
- An `epic-summary.md` in the same folder
- Access to the Atlassian MCP server (`mcp_atlassian_*` tools)
- The target Jira project key (e.g., `TI`)

---

## Step 0: Identify the epic folder

The epic folder contains `epic-summary.md` and story files. It is one of:

- **Phase folder:** `workspace/projects/<slug>/eng/<phase>/` where `<phase>` is `alpha`, `beta`, `ga`, or `fast-follow` (see `docs/WORKSPACE-PHASES.md`)
- **Single epic folder:** `workspace/projects/<slug>/eng/epic/` or `workspace/projects/<slug>/epics/` (legacy)

Ask the user **which phase** (or epic folder) to push if the project has multiple phase folders. Use that folder as the working directory for this run.

---

## Step 1: Read Reference Materials

Read the shared conventions:
- `standards/conventions.md` — Cross-referencing rules (Jira tickets > titles > never story numbers)

---

## Step 2: Identify Scope

Ask the user what to push:

1. **Full epic** — Push all stories from the epic
2. **Single story** — Push one specific story file
3. **Subset** — Push selected stories by title

Then read the relevant files:
- `epic-summary.md` — For epic-level metadata
- The story file(s) to push

---

## Step 3: Resolve the Target Jira Project

1. Use `mcp_atlassian_getAccessibleAtlassianResources` to get the cloud ID
2. Use `mcp_atlassian_getVisibleJiraProjects` to find the target project
3. Use `mcp_atlassian_getJiraProjectIssueTypesMetadata` to identify available issue types (Epic, Story, Task, Bug, etc.)

---

## Step 4: Check for Existing Epic in Jira

Before creating anything, search for an existing epic:

1. Use `mcp_atlassian_searchJiraIssuesUsingJql` with JQL:
   ```
   project = {KEY} AND issuetype = Epic AND summary ~ "{epic title}"
   ```
2. If an epic exists:
   - Confirm with the user: "Found existing epic {KEY}-{ID}: '{title}'. Add stories to this epic?"
   - If yes, use this epic as the parent
3. If no epic exists:
   - Create one using `mcp_atlassian_createJiraIssue` with:
     - `issueTypeName: "Epic"`
     - `summary`: Epic title from `epic-summary.md`
     - `description`: Epic description + PRD requirements covered

---

## Step 5: Push Stories to Jira

For each story to push:

### 5.1 Check if story already exists

Search Jira for the story title:
```
project = {KEY} AND summary ~ "{story title}" AND issuetype = Story
```

- If found: **Update** the existing issue (don't create a duplicate). Confirm with user first.
- If not found: **Create** a new issue.

### 5.2 Create/Update the Jira issue

Use `mcp_atlassian_createJiraIssue` (or `mcp_atlassian_editJiraIssue` for updates) with:

| Field | Source |
|-------|--------|
| `projectKey` | Target project |
| `issueTypeName` | `"Story"` |
| `summary` | Story title (without the `Story N:` prefix and `[Tag]` prefix — those become labels) |
| `description` | Full story content in markdown: User Story, Context, Technical Details, Acceptance Criteria, Story Points, Cursor Prompt |
| `parent` | Epic key from Step 4 |

### 5.3 Add labels

Add labels based on the story tag:
- `[BE]` → label: `backend`
- `[FE]` → label: `frontend`
- `[FF]` → label: `feature-flag`
- `[QA]` → label: `qa`
- `[SEC]` → label: `security`
- `[DB]` → label: `database`
- `[LOG]` → label: `observability`
- `[ANALYTICS]` → label: `analytics`
- `[RESEARCH]` → label: `spike`
- `[OPS]` → label: `operations`
- `[DOCS]` → label: `documentation`
- `[PERF]` → label: `performance`
- `[ALERT]` → label: `alerting`

### 5.4 Record the mapping

After creation, record the mapping between the local story and the Jira ticket:

```markdown
| Local File | Jira Ticket | Title |
|------------|-------------|-------|
| story-1-... | TI-3850 | Conversational Search UI |
```

---

## Step 6: Update Cross-References

After all stories are created and Jira ticket IDs are known:

1. **Update local story files** — Replace story title references in `Depends on:` and AC sections with Jira ticket IDs
2. **Update `epic-summary.md`** — Add a Jira Ticket column to the stories table
3. **Update Jira descriptions** — If any story's description references other stories by title, update those references to use the now-known Jira ticket IDs

Use `mcp_atlassian_editJiraIssue` for Jira updates and file editing tools for local files.

---

## Step 7: Create Issue Links in Jira

For stories with dependencies:

1. Read the `Depends on:` field from each story
2. Use `mcp_atlassian_jiraWrite` with `action: "createIssueLink"` to create "Blocks" links:
   - If Story B depends on Story A: A **blocks** B
3. Link type: Use `mcp_atlassian_jiraRead` with `action: "getIssueLinkTypes"` to find the correct link type name

---

## Step 8: Rename folder and files (if in workspace/projects/)

If the epic folder is under **`workspace/projects/`** (and not already prefixed with a Jira key):

1. **Rename the epic folder** to **`{EPIC_KEY}-{current-slug}`** (e.g. `TI-3556-endpoint-catalog-stories`). Use the epic key returned from Jira (Step 4).
2. **Rename each story file** in that folder to **`{STORY_KEY}-{original-filename}`** (e.g. `story-1-aeo-pages.md` → `TI-3927-story-1-aeo-pages.md`). Use the story key from the mapping (Step 5.4).
3. **Do not rename** `epic-summary.md` unless you adopt a convention to prefix it with the epic key (e.g. `TI-3556-epic-summary.md`). Optional: leave as `epic-summary.md` for consistency with other tools.

After renaming, update any internal references (e.g. in epic-summary.md) that point to the old file names.

---

## Step 9: Output Summary

Present to the user:

1. **Created issues** — Table of all Jira tickets created with links
2. **Updated issues** — Any existing issues that were updated
3. **Dependencies linked** — Which issue links were created
4. **Cross-references updated** — Which local files were updated with Jira IDs
5. **Folder/files renamed** — If in workspace/projects/, new folder and file names with Jira key prefix
6. **Skipped** — Any stories not pushed (with reason)

---

## Modes of Operation

### Full Epic Push
```
/push-to-jira for workspace/projects/c2-conversational-search/epics/
```
Pushes the epic + all stories.

### Single Story Push
```
/push-to-jira story-14-e2e-test-suite.md to TI project
```
Pushes one story to an existing or new epic.

### Update After Changes
```
/push-to-jira update workspace/projects/c2-conversational-search/epics/
```
Compares local files to Jira and updates any that have changed.
