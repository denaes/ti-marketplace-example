# Cursor Prompt Builder Skill

## 1. Purpose

Generate a **Cursor IDE prompt** for each story that gives an engineer everything they need to start coding. The prompt must be specific enough that Cursor can locate the right files, understand the patterns, and produce correct code on the first attempt.

## 2. When This Skill Is Used

Every story file generated by the **write-epics-and-stories-from-prd** workflow (or `/write-epics-and-stories-from-prd`) must include a `## Cursor Prompt` section produced using this skill. The only exceptions are `[RESEARCH]` stories (which produce documents, not code).

## 3. Prompt Structure

Every Cursor Prompt must follow this exact structure:

```
Role: [Specific role and tech stack context]

Task: [One sentence describing what to implement, referencing the story title]

CRITICAL — Read these docs FIRST (in this order):
1. [Most important doc — architecture/flow doc relevant to this story]
2. [Second doc — patterns or conventions to follow]
3. [Third doc — if needed]

CODEBASE CONTEXT:
[Numbered list of existing patterns, files, modules, and DI tokens to reference.
Each item must include the file path and a brief explanation of what it does
and why it's relevant to this story.]

WHAT TO IMPLEMENT:
[Step-by-step breakdown into Parts. Each Part is a logical unit of work:]

Part 1 — [Component/Layer]:
- [Specific implementation step with file path]
- [What to create/modify]
- [Code pattern to follow]

Part 2 — [Component/Layer]:
- [...]

ARCHITECTURE RULES:
[Key constraints from the codebase — DI patterns, module boundaries, etc.]

DO NOT:
[Explicit anti-patterns and things to avoid — at least 3 items]

TESTING:
[Specific test scenarios to verify — not generic, tied to this story's ACs]
```

## 4. Prompt Generation Rules

### 4.1 Role Line

- Be specific: "Senior Backend Engineer working on a NestJS V3 app" not "Developer"
- Include the tech stack: framework, language, key libraries
- If fullstack, state which layers are in scope

### 4.2 CRITICAL Docs

- Order matters — most important first
- Only include docs that **exist in the repo** (verify via codebase search)
- Max 4 docs — more than that means the task is too broad
- Reference specific sections within docs when possible (e.g., "Section 3: Event Flow")

### 4.3 Codebase Context

This section must be **derived from actual search** of the codebase (`ti/`, `ti/v3/`) so the user and Cursor see **current state** — what exists today, which patterns and DI tokens are used, and what the architecture allows. Do not invent paths or patterns; verify by search. Proposals in the prompt must **respect existing architecture and rules** (e.g. `ti/v3/docs/ARCHITECTURE_VALIDATOR.md`). See `standards/conventions.md` (Codebase-first).

It must contain:

1. **Existing patterns to follow** — "The FeatureFlagAdapter uses hexagonal architecture. Inject via `@Inject(FEATURE_FLAG_ADAPTER)`."
2. **Specific file paths** — Absolute from workspace root: `v3/src/modules/chat/chat.controller.ts`
3. **DI tokens and interfaces** — If the codebase uses dependency injection, list the tokens
4. **Database schemas** — If the story touches data, list the relevant table/entity files
5. **Event types** — If the story uses events, list existing event types and how to emit/listen

### 4.4 What To Implement

- Break into **Parts** (1 Part per logical unit of work)
- Each Part should be independently verifiable
- Include specific file paths for where to create/modify code
- Reference the pattern to follow (e.g., "Follow the same pattern as `VectorSearchController.search()`")
- For frontend: include component names, props, state structure
- For backend: include endpoint paths, request/response shapes, service methods

### 4.5 DO NOT

Always include at least 3 anti-patterns. Common ones:

- "Do not modify files outside your scope" (e.g., "Do not modify any V3 backend code")
- "Do not use `@ts-ignore` or `eslint-disable`"
- "Do not call HTTP endpoints when the system uses events"
- "Do not import from the concrete implementation — use the DI token"
- "Do not hard-code values — use environment variables or config"

### 4.6 Testing

- Tied to this story's acceptance criteria — not generic
- Include both positive and negative test cases
- For backend: specify exact HTTP requests and expected responses
- For frontend: specify user interactions and expected DOM state

## 5. Story-Type Variations

| Story Type | Prompt Emphasis | Special Sections |
|-----------|----------------|------------------|
| `[BE]` | File paths, DI tokens, event types, API contracts | Request/response schemas in CODEBASE CONTEXT |
| `[FE]` | Component tree, state management, API consumption | UI mockup references, styling framework notes |
| `[FF]` | Guard placement, flag checks, Redis TTL | Both backend and frontend sections |
| `[DB]` | Migration files, entity definitions, index strategy | Schema before/after in CODEBASE CONTEXT |
| `[SEC]` | Access control logic, auth guards, data filtering | Threat model reference in CRITICAL docs |
| `[QA]` | Test framework, fixtures, Page Object Model | See `qa_test_planner` skill for QA-specific prompt |
| `[LOG]` | Logging library, structured log format, metric names | Existing log patterns in CODEBASE CONTEXT |
| `[ANALYTICS]` | Event schema, tracking targets, correlation IDs | Analytics SDK usage patterns |
| `[OPS]` | Admin endpoints, guard checks, idempotency | Existing admin patterns |
| `[RESEARCH]` | **No Cursor Prompt needed** — output is a document, not code | Replace with "## Research Output" section |

## 6. Quality Checklist

Before finalizing a Cursor Prompt, verify:

- [ ] Every file path referenced actually exists in the codebase (or is clearly marked as NEW)
- [ ] The "Read these docs FIRST" docs exist and the referenced sections are real
- [ ] DI tokens, interface names, and class names are spelled correctly
- [ ] The Parts are ordered logically (dependencies first)
- [ ] The DO NOT section includes at least 3 items specific to this story
- [ ] The TESTING section maps to the story's acceptance criteria
- [ ] No references to story numbers — only Jira tickets or story titles (see `standards/conventions.md`)
