# Jira Ticket Context Skill

## 1. Purpose

When an engineer is working from a Jira ticket (e.g. assigned to TI-XXXX), this skill describes how to **fetch the ticket** and **parse its description** into structured implementation context: acceptance criteria, technical notes, Cursor prompt, and test plan.

## 2. When This Skill Is Used

- The user says "start work on TI-XXXX", "work on TI-3927", "I'm assigned to TI-3516", or similar.
- Any workflow or agent that needs to load a Jira issue to drive implementation.

## 3. Fetching the Issue (Atlassian MCP)

1. **Cloud ID:** Call `mcp_atlassian_getAccessibleAtlassianResources` and use the returned `cloudId` for Jira calls.

2. **Get issue by key:**
   - Tool: `mcp_atlassian_getJiraIssue`
   - Parameters: `issueIdOrKey`: e.g. `TI-3516`
   - Request at least: `summary`, `description`, `status`, `priority`, `story_points`, `parent`, `labels`, `assignee`
   - Optional: `expand: "renderedFields"` if the description is in Atlassian Document Format (ADF) and you need rendered HTML/markdown.

3. **If the issue is not found:** Check key format (project key + hyphen + number). Confirm the user has access to the project.

## 4. Parsing the Description

Jira descriptions often follow the structure from **Jira Architect** (`skills/engineering-management/jira_architect/SKILL.md`). Look for these sections (headers or emoji markers):

| Section | Common markers | What to extract |
|--------|----------------|-----------------|
| Acceptance criteria | `## Acceptance Criteria`, `🎯 Acceptance Criteria`, bullet/list items | List of testable outcomes; use as definition of done |
| Technical implementation | `## Technical Implementation Notes`, `🛠`, `## Technical Details` | Scope, file/module hints, constraints |
| Test plan | `## Test Plan`, `🧪` | Manual or automated test steps |
| Cursor prompt | `## CURSOR DEV PROMPT`, `## Cursor Prompt`, `🤖`, fenced code block | Full Cursor IDE prompt — primary implementation instructions |

- **Wiki/ADF:** If description is ADF JSON or Wiki markup, use the **rendered** content when available so headings and lists are clear.
- **Fallback:** If sections are missing, treat the whole description as context and infer ACs from any bullet lists.

## 5. Linking to Local Story Files

Stories generated by the tech lead may exist under `workspace/projects/<slug>/epics/` or `workspace/_references/story-examples/`:

- **Search by Jira key:** e.g. `TI-3516` in `workspace/projects/**/*.md`
- **Search by title:** story files often have the same title as the Jira summary

If a local `story-*.md` (or `TI-XXXX-story-*.md`) exists, prefer it for the **Cursor prompt** and **ACs** — it is usually more detailed than the Jira description. Still show Jira status and link.

## 6. Output for the Engineer

Provide:

1. **Ticket:** key, summary, status, assignee
2. **Acceptance criteria** (bulleted)
3. **Technical scope** (short summary: backend/frontend/QA, key files or areas)
4. **Cursor prompt** (full or "Read these docs FIRST" + "WHAT TO IMPLEMENT" + "DO NOT")
5. **Jira link:** `https://thoughtindustries.atlassian.net/browse/{KEY}` (or project base URL)

Cross-references in any written output must use **Jira ticket IDs** per `standards/conventions.md`.
